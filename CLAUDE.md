# CLAUDE.md — GymApp (Telegram Mini App)

> Этот файл читается агентом первым в каждой сессии. Он задаёт правила игры,
> команду проверки (gate) и формат, в котором ему ставят задачи.
> Полная спецификация проекта — в `SPEC.md`. Этот файл её НЕ заменяет, а дополняет
> правилами для агентной работы.

---

## 0. Как ты (агент) работаешь над этим проектом

1. **Сначала читаешь `SPEC.md`** — это источник истины по архитектуре, типам, фазам.
2. Работаешь строго в рамках **одной фазы за раз** (см. раздел «Фазы» в SPEC.md).
   Не забегай вперёд: не пиши код фазы 5, когда задача — фаза 2.
3. **После КАЖДОГО логического куска** прогоняешь gate (см. ниже) и не считаешь
   работу сделанной, пока gate не зелёный.
4. Если что-то в спеке противоречиво или недосказано — **останавливаешься и
   спрашиваешь**, а не додумываешь молча. Зафиксируй вопрос явно.
5. Никогда не трогаешь файлы вне согласованной зоны задачи (см. «Границы»).

## 1. Gate — команда проверки готовности

**Ничего не считается готовым, пока это не проходит:**

```bash
npx tsc --noEmit && npm run lint && npm test
```

> ⚠️ Скрипты `lint` / `test` появляются по мере настройки проекта (фаза 1).
> До их добавления gate = `npx tsc --noEmit`. По мере роста проекта добавляй сюда
> сборку (`npm run build`) как финальную проверку перед коммитом фазы.

**Правило:** если gate красный — чинишь, пока не зелёный. Не отдаёшь «почти готово».
Не отключаешь правила линтера и не глушишь ошибки типов через `any` / `@ts-ignore`,
чтобы gate прошёл — это считается провалом задачи.

## 2. Жёсткие правила проекта (из SPEC.md, не нарушать)

1. **Дата-слой:** компоненты ходят за данными только через TanStack Query поверх
   `services/*`. Сервисы асинхронны (`Promise`) уже на mock-этапе. Прямых обращений
   к mock-данным из компонентов быть не должно.
2. **Типы — в `types/index.ts`**, zod-схемы — в `schemas/`. Не плодить дубли типов.
3. **Цикл программы** — свойство `Program` (`cycleLength`), прогресс юзера — в
   `UserProgramState`. `currentDayIndex` двигается ТОЛЬКО при явном `finishWorkout`,
   по формуле `(i + 1) % cycleLength`.
4. **Активная тренировка** — Zustand с `persist`, обязательное восстановление сессии
   при маунте. Источник истины по завершённым тренировкам — `services/history`, не стор.
5. **Ключи AI-провайдеров — только серверные.** `services/ai.ts` ходит во внутренние
   `/api/ai/*`. Никаких ключей в браузерном коде.
6. **Mobile-first**, max-width 430px. Обе темы через Tailwind `dark:`.
7. **Safe Area Insets** через `useSafeAreaInsets()`, отступы не хардкодить.
8. **Маленькие компоненты**, одна ответственность. Страницы в `app/` — тонкие,
   только собирают компоненты из `components/`.
9. **Mock-данные реалистичные**, на русском, реальные названия упражнений.

## 3. Границы задачи (что можно и нельзя трогать)

- По умолчанию агент меняет файлы **только в рамках текущей фазы**.
- НЕ редактирует `SPEC.md` и `CLAUDE.md` без явного указания.
- НЕ устанавливает новые зависимости, кроме перечисленных в стеке SPEC.md, без
  явного согласования.
- НЕ удаляет существующий рабочий код «заодно». Рефакторинг — отдельная задача.

## 4. Рабочий цикл по фазам (git как обвязка)

Одна фаза = одна ветка. Поток:

```
1. Создать ветку:        git checkout -b phase-N-<краткое-имя>
2. Поставить задачу агенту по шаблону ниже
3. Агент пишет код + прогоняет gate до зелёного
4. Ты ревьюишь diff:     git diff main
5. Мёржишь:              git checkout main && git merge phase-N-...
6. Следующая фаза — новая ветка
```

Коммиты — маленькие и осмысленные, по логическим кускам внутри фазы, а не один
гигантский коммит в конце.

## 6. Первая задача для практики (рекомендация)

Начни с **фазы 1 + 2** как единой первой агентной задачи:

- Фаза 1: инициализация (Next.js 15, Tailwind, shadcn, Zustand+persist,
  TanStack Query, Telegram SDK) + настройка `tsc`/`lint`/`test`, чтобы gate заработал.
- Фаза 2: типы из SPEC.md в `types/index.ts`, zod-схемы в `schemas/`, async-сервисы
  над mock, Query-хуки.

Почему именно эти две: они изолированы, результат проверяется компилятором почти
полностью, и в процессе ты настроишь сам gate. Это лучшая «учебная» петля —
дёшево, быстро, объективно проверяемо. Цель первой задачи — не столько код,
сколько прочувствовать цикл «задача → работа агента → gate → ревью → мёрж».

## 7. Когда переходить от Claude Code к своему harness (Agent SDK)

Не раньше, чем поймаешь конкретное повторяющееся действие, которое хочешь
автоматизировать без своего участия. Типичный триггер: «я устал руками запускать
одну и ту же фазу-задачу и проверять gate — хочу скрипт, который гоняет агента в
цикле, пока gate не зелёный, и сам останавливается». Вот тогда — Agent SDK, и
переносишь в код именно этот понятный процесс, а не выдуманный.
